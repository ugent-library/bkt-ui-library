Forms & Inputs

Forms are where the "opinions over options" principle is most testable. Every field is a question. Questions cost the user something. Ask only when necessary, from the right person, once. The form's job is to make answering easy — not to be thorough.

Accessibility rules for forms

  • Every <input> must have an associated <label> — never use placeholder as the only label
  • Group related controls with <fieldset> and <legend>
  • Error messages must be associated via aria-describedby or rendered adjacent to the field
  • autocomplete attributes on personal data fields (name, email) improve usability and are WCAG 2.1 AA required
  • Required fields: use required attribute, not just a visual asterisk
  • Never disable a submit button to indicate errors — show the errors instead
  • On server-side validation failure (HTMX form post), move focus to the first error or the error summary
html — accessible error association
<div class="mb-3">
  <label for="email" class="form-label">Email address</label>
  <input type="email"
    class="form-control is-invalid"
    id="email"
    aria-describedby="email-error"
    autocomplete="email"
    required>
  <div id="email-error" class="invalid-feedback">
