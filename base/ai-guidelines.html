AI Code Guidelines & Testing Protocol

This document governs how AI-assisted code — including code written by Claude in collaboration with this design system's maintainer — is written, reviewed, and verified. AI systems produce plausible-looking code that contains real errors in predictable categories. Knowing those categories is the correction.

The core problem

Human developers make random mistakes. AI systems make systematic, confident-sounding mistakes in the same categories every time. You cannot review AI code the way you review human code. You need specific checkpoints, run in order.

AI is systematically unreliable for

  • ARIA attributes that look right but aren't wired up
  • Your specific CSS class names — it invents plausible ones
  • HTMX attribute combinations and timing edge cases
  • Real states: empty, error, loading, locked, concurrent edit
  • Copy and microcopy — defaults to generic American English
  • Bootstrap version differences (5.3 vs 5.2)
  • Whether a pattern exists in your system vs inventing one
  • Focus management after dynamic content updates
  • Right-to-left text handling

AI is reliably good for

  • Scaffolding structure and layout
  • Bootstrap utility class combinations
  • schema.org JSON-LD structure
  • HTMX attribute names and basic patterns
  • Refactoring repetitive HTML consistently
  • Catching typos and inconsistencies
  • Explaining what a pattern should do before writing it
  • Generating realistic, varied placeholder content

Testing protocol — run these in order after every AI-generated component

Run in this order. Structure before detail before interaction before copy.
1

Load in browser — visual inspection

Open the page. Resize to 375px. Is the spacing consistent? Is anything overflowing, clipping, or misaligned?

Common AI failure: correct structure but wrong spacing, margins collapsing incorrectly, elements not on the 4px grid.
2

Heading hierarchy

Open DevTools → Elements. Is there exactly one <h1>? Does h1 → h2 → h3 make sense without CSS? Install the Headings Map extension and look at the outline.

Common AI failure: uses heading tags for visual sizing, or creates duplicate h1/h2 when the shell already has one.
3

Keyboard navigation — Tab through everything

Put the mouse away. Tab through every interactive element. Is every button and link reachable? Is the focus order logical? Is there a visible focus ring? Can you activate with Enter / Space? Do modals trap focus correctly?

Common AI failure: tabindex="-1" on focusable elements, or focus rings removed with outline: none with no replacement.
4

Form label association

Click each label text. Does it focus the correct input? Check: every <input> has id matching its <label for="...">. Check: validation errors linked via aria-describedby.

Most common AI failure: generates correct label text but wrong or missing for attribute. The input and label look associated but aren't.
5

Screen reader smoke test

macOS: Cmd + F5 (VoiceOver). Navigate by headings (H), landmarks (W), form controls (F). Ask: are landmarks correct? Do icon-only buttons have names? Are status messages announced? Does "● OA" announce as "Open access"?

Common AI failure: icon buttons without accessible names, aria-live on wrong element, decorative icons not hidden with aria-hidden="true".
6

Colour contrast

DevTools → Lighthouse → Accessibility. WCAG 2.1 AA minimum: 4.5:1 for body text, 3:1 for large text and UI elements. Check: badge text on pale backgrounds, grey helper text, disabled states, text on --bt-blue-900 background.

Note: --bt-gray-700 (#6b6660) on white = 5.1:1 ✓. --bt-gray-300 (#b5b2ae) on white = 2.8:1 ✗ — decorative use only.
7

JavaScript disabled — progressive enhancement baseline

DevTools → Settings → Disable JavaScript. Reload. Do forms still submit to their action URL? Do links still navigate? Is there a meaningful page state?

HTMX is progressive enhancement. A form without JavaScript should post to a real URL and return a full-page response. If it only works with JS, fix the HTML.
8

HTMX dynamic interactions

Re-enable JavaScript. For each HTMX interaction: does the loading indicator appear? Does the target update? Does the URL update when expected? After a swap, is focus in the right place? Does Back work after hx-push-url?

9

Edge cases — the states AI always forgets

For every list and form, explicitly test:

  • Empty: 0 items. Is there a blank state?
  • Error: server-side validation failure. Are errors visible and useful?
  • Long strings: paste a 300-character title. Does it overflow?
  • Loading: does the loading state look intentional, not broken?
  • No permission: what does a locked or restricted record look like?
  • RTL: paste Arabic text. Does it render with correct direction?
10

Copy review — is this your voice?

Read every label, button text, error message, and empty state out loud. Check: is it "research output" not "publication"? Does the error say what went wrong AND what to do? Is the button a verb? Is the tone correct for this audience?

AI defaults to American English corporate tone. The Biblio voice is direct, institutional, and precise — but not cold. "Your research output has been received" not "Successfully submitted!"

Pre-commit checklist — copy into PR description

markdown
## Accessibility & quality checklist

### Structure
- [ ] Single <h1> per page, correct heading hierarchy
- [ ] Semantic HTML: article, section, nav, aside, main
- [ ] Landmark regions present: header, nav, main
- [ ] All interactive elements reachable by Tab, logical order

### Forms
- [ ] Every input has an associated <label> (click label → input focused)
- [ ] Error messages linked via aria-describedby
- [ ] Required fields use the `required` attribute
- [ ] Checkboxes/radios grouped with fieldset + legend

### Icons
- [ ] Decorative icons have aria-hidden="true"
- [ ] Icon-only buttons have accessible name (aria-label or .visually-hidden span)

### Dynamic content (HTMX)
- [ ] Dynamic regions wrapped in aria-live="polite"
- [ ] Loading indicators have role="status" and aria-label
- [ ] Focus managed after swaps
- [ ] Forms work with JavaScript disabled

### Visual & content
- [ ] Tested at 375px width — no overflow
- [ ] Empty state handled
- [ ] Error state handled with correct alert variant
- [ ] Copy reviewed: "research output" not "publication"
- [ ] Error messages say what went wrong + what to do
- [ ] Button labels are verbs

### Design system
- [ ] Uses existing design system classes, no invented names
- [ ] Uses var(--bt-*) tokens, not raw hex values

Known gaps — tracked here, not hidden

Open Icon system

The original UI kit uses if if-* (UGent icon font). This system uses Remix Icons (ri-* via CDN). Decision needed before production: adopt Remix Icons system-wide, or bring in the UGent font.

Open Self-hosted fonts

IBM Plex currently loads from Google Fonts CDN. For production, fonts must be self-hosted. assets/ibm-plex/ exists but is empty.

Open Dutch language voice

All templates are in English. Writing guidelines and tone of voice for Dutch are not documented. Error messages and empty states particularly need Dutch equivalents.

Planned Depotmanagement template

Dense operational interfaces for depot staff (task detail, shelf management, scanning) need their own template with higher information density defaults.

Planned Error pages (404, 500)

Should live in templates/. A library's 404 should be warm and helpful. The existing prototype has these — they need bringing into this system.

Planned Writing guidelines

The design system enforces visual consistency but not verbal consistency. A foundations/writing.html page documenting button label rules, error message patterns, and voice is needed.
