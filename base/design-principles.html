Design Principles

Four principles. Not guidelines, not suggestions — decision tools. When two directions compete, these are what you use to choose. If a principle wouldn't help you decide between two real options, it isn't doing its job.

01

Structure is the style

The Boekentoren is beautiful because its logic is exposed, not hidden. Our interfaces work the same way: a rigorous system of spacing, type, and hierarchy generates visual character without ornament. Internal complexity — between our systems, our teams, our workflows — never becomes the user's problem.

The test

Remove colour from this screen. Does hierarchy, relationship, and purpose still read clearly? If not, fix the structure first.

✓ Do

  • Use whitespace to create hierarchy
  • Let type scale carry the visual weight
  • Question "why is this decorated"

✗ Don't

  • Add colour where structure isn't working
  • Expose internal system seams to users
  • Use motion to compensate for poor hierarchy
02

Opinions over options

We make the hard decisions so the user doesn't have to. An interface full of options is an interface that has failed to think. We ask only for what we genuinely need, only once, only from the person who actually owns that information. A researcher's title capitalisation is the researcher's decision. A curator's metadata schema is the curator's domain. The interface enforces those boundaries structurally.

"Een keer informatie vragen is plagen, twee keer is pesten."

The test

For every field, modal, or decision point: whose job is this to decide? If it's ours, we've already decided. If it's the user's, we ask once and remember. If it belongs to someone else entirely, it shouldn't be here. And: does this screen reveal any of our internal organisational complexity to someone who shouldn't need to care about it?

03

The past and the future share the same grid

Our collection spans centuries. Our researchers work at the edge of what's known. The design system holds both without favouring either. It doesn't look like a startup and it doesn't look like a museum. A 15th century manuscript and a 2024 preprint should feel equally at home in every component we build.

We follow standards not out of obligation but because standards serve knowledge sharing and interoperability. When a standard helps knowledge travel further, we use it.

The test

Put a 15th century manuscript and a 2024 open access preprint in the same component. Do both feel equally at home? And: if we refresh the visual layer in five years, does the structure underneath still hold?

04

Knowledge wants to move

Our job is not to store knowledge — it's to make it travel. Every design decision asks: does this make it easier to find, understand, share, and build on? We remove friction from the path between a researcher's work and the world that needs it.

We also design for the unexpected connection — the discovery that changes how someone thinks. We don't just answer the question asked. We create the conditions for better questions. And what we build and document ourselves, we share openly — inside and outside the university.

The test

How many steps from finding something to sharing or citing it? That number should always be shrinking. And from any item: can a curious person find something they didn't know they were looking for?

When principles conflict

Principles are useful precisely because they create productive tension. Here's how to navigate the common conflicts:

01 vs 02 — Structure says remove it. Expertise needs it.

Principle 02 wins for expert users when the information genuinely belongs to their workflow. But first ask: is there a structural way to surface it that doesn't add visual noise for everyone? Progressive disclosure is usually the answer — present the simple path first, and let experts find the depth.

02 vs 04 — We've decided. But should the user have this choice?

When the choice affects knowledge dissemination — open access settings, sharing permissions, visibility — give it to the user. These are genuinely theirs. When the choice is about internal presentation or metadata formatting, it's ours. We've already decided.

03 vs 01 — Heritage items need special treatment. But that breaks consistency.

Resist special treatment. The system's job is to be a neutral container that dignifies every record equally. If a manuscript needs more context than a preprint, that's a content problem — solve it with good metadata and description, not a different component.
