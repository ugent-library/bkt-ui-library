Using in your app

How to integrate the Booktower design system into an application — CSS, icons, surfaces, and overrides.

What you need

The design system ships as two things: a compiled CSS file and a font file. That's it.

Files to copy from the UI kit
assets/booktower.css        ← the design system
 assets/fonts/icon-font.woff2  ← icon font (primary)
 assets/fonts/icon-font.woff   ← icon font (fallback)

The font files must be served at /assets/fonts/ relative to your app root, or you'll need to update the font paths (see below).

Load order

Bootstrap must come before booktower.css. The design system overrides and extends Bootstrap — it does not replace it.

HTML <head>
<link rel="stylesheet" href="bootstrap.min.css">
<link rel="stylesheet" href="booktower.css">

For Bootstrap JS (dropdowns, modals etc.):

Before </body>
<script src="bootstrap.bundle.min.js"></script>

Bootstrap CDN links if you're not bundling it:

CDN
<!-- CSS -->
<link rel="stylesheet"
  href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.3/dist/css/bootstrap.min.css">
<link rel="stylesheet" href="/assets/booktower.css">

<!-- JS -->
<script
  src="https://cdn.jsdelivr.net/npm/bootstrap@5.3.3/dist/js/bootstrap.bundle.min.js">
</script>

Surfaces

Add data-surface="public" or data-surface="backoffice" to your <body> or any layout container. This activates the correct typographic personality for the context — serif headings and generous spacing for public-facing pages, sans and compact for backoffice tools.

Public app
<body data-surface="public">
Backoffice app
<body data-surface="backoffice">

You can mix surfaces within a page — for example, a public-facing header above a backoffice data entry area:

Mixed surfaces
<header data-surface="public">...</header>
<main data-surface="backoffice">...</main>

Icons

Icons are included in booktower.css — no extra stylesheet needed. The font files must be reachable at /assets/fonts/.

See the icons page for the full list of available icons.

If your font path is different

The icon font is referenced as /assets/fonts/icon-font.woff2 in booktower.css. If your app serves static files from a different path, you have two options:

  1. Serve the fonts at the expected path. This is the simplest option — put the font files at /assets/fonts/ on your server.
  2. Override the font-face declaration in your own stylesheet, loaded after booktower.css:
    your-overrides.css
    @font-face {
  font-family: "icon-font";
  src: url("/your/path/icon-font.woff2") format("woff2"),
       url("/your/path/icon-font.woff") format("woff");
}

Overrides

Keep overrides minimal and in one place — a single overrides.css loaded after booktower.css. Use CSS custom properties where possible, since most visual decisions are already tokenised.

Typical override pattern
/* overrides.css — loaded after booktower.css */

/* Example: change the primary brand colour for this app */
:root {
  --bt-blue: #0057a8;
  --bt-dark-blue: #003d7a;
}

Avoid overriding component classes directly where a token will do the same job. The token system is the intended extension point.

Keeping up to date

When the UI kit updates, copy the new booktower.css and font files into your app and test. There is no automated update mechanism yet — this is intentional while the system is still evolving.

Breaking changes will be noted in the CHANGELOG. In practice, token-based overrides are very unlikely to break between updates.
