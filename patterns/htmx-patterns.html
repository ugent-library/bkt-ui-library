HTMX & Hypermedia Patterns

The Biblio stack is SSR-first with HTMX for interactions. This page documents the patterns your developers should reach for — and the rules that keep the architecture clean. HTMX works best when you treat it as what it is: an extension of HTML, not a JavaScript framework with different syntax.

The architecture in one sentence: the server renders HTML, HTMX swaps fragments of it, and progressive enhancement means every form and link works without JavaScript first. A page should be functional without HTMX and better with it — not broken without it.

Core principles for this stack

HTML first

Every form posts to a real URL. Every link navigates. HTMX intercepts and upgrades — it never replaces the underlying semantic behaviour. Test with JavaScript disabled regularly.

Server owns state

The server returns HTML fragments, not JSON for the client to render. HTMX swaps them in. This means your templates stay on the server, close to your data.

Stable URLs

Every significant view has a URL. Detail pages, search results with filters, paginated lists — all bookmarkable and shareable. Use hx-push-url when a user would want to bookmark the current state.

Accessible by default

HTMX swaps don't automatically notify screen readers. You must use aria-live regions for dynamic updates, and manage focus explicitly when content changes. See the a11y section below.

Pattern 1 — Live search

Used in: backoffice overview, public search, people lookup. Debounced keyup, targets a results container, shows a loading indicator.

The pattern
Type to search…
html
<!-- The input -->
<label for="q" class="visually-hidden">Search research output</label>
<input type="search"
  class="form-control"
  id="q"
  name="q"
  placeholder="Search by title, author, DOI…"
  aria-controls="search-results"
  hx-get="/search"
  hx-trigger="keyup changed delay:300ms, search"
  hx-target="#search-results"
  hx-swap="innerHTML"
  hx-indicator="#search-spinner">

<!-- Loading indicator -->
<span id="search-spinner" class="htmx-indicator">
  <span class="spinner-border spinner-border-sm" role="status" aria-label="Searching…"></span>
</span>

<!-- Live region — screen readers announce updates -->
<div id="search-results"
  role="region"
  aria-live="polite"
  aria-label="Search results">
  <!-- Server returns article elements here -->
</div>
html — server partial response (/search?q=urban)
<!-- This is what the server returns — raw HTML fragment, no shell -->
<p class="text-muted">12 results for "urban"</p>

<article class="card card-research">
  <div class="card-body">
    <h2 class="card-title">
      <a href="/research/01K9...">Urban forests as essential infrastructure…</a>
    </h2>
    <p class="text-muted">Esperon-Rodriguez et al. · 2026</p>
  </div>
</article>

<!-- More articles… -->

Pattern 2 — Inline form save (draft autosave)

Used in: deposit flow. The form posts on blur or button click. The server returns a success fragment or validation errors. No page reload.

Save with status feedback
html
<form hx-post="/deposit/01K9.../save"
  hx-target="#save-feedback"
  hx-swap="innerHTML"
  hx-indicator="#save-spinner">

  <div class="mb-3">
    <label for="title" class="form-label">Title</label>
    <input type="text" class="form-control" id="title" name="title"
      value="{{ publication.title }}">
  </div>

  <!-- More fields… -->

  <div class="d-flex gap-2 align-items-center">
    <button type="submit" class="btn btn-secondary btn-sm">
      Save draft
      <span id="save-spinner" class="htmx-indicator ms-1">
        <span class="spinner-border spinner-border-sm" role="status" aria-label="Saving…"></span>
      </span>
    </button>
    <span id="save-feedback" role="status" aria-live="polite"></span>
  </div>
</form>

<!-- Server returns on success (replaces #save-feedback) -->
<span>
  <i class="ri-checkbox-circle-line" aria-hidden="true"></i>
  Saved
</span>

<!-- Server returns on validation error (replaces whole form via hx-target="closest form") -->
<!-- Return the full form HTML with is-invalid classes applied to failing fields -->

Pattern 3 — Inline confirm (destructive actions)

Used in: delete record, remove author, archive item. Show a confirmation in-place before the irreversible action. Never use a modal for a single-field confirmation.

In-place confirm — no modal, no page reload
Urban forests as essential infrastructure…
Delete this record?
html — the HTMX version
<!-- Initial state -->
<div id="record-row-{{ id }}">
  <span>{{ title }}</span>
  <button
    hx-get="/research/{{ id }}/delete-confirm"
    hx-target="#record-row-{{ id }}"
    hx-swap="outerHTML"
    class="btn btn-ghost btn-sm"
    aria-label="Delete {{ title }}">
    <i class="ri-delete-bin-line" aria-hidden="true"></i>
  </button>
</div>

<!-- Server returns (replaces the whole row) -->
<div id="record-row-{{ id }}">
  <span>{{ title }}</span>
  <span>Delete this record?</span>
  <button
    hx-delete="/research/{{ id }}"
    hx-target="#record-row-{{ id }}"
    hx-swap="outerHTML swap:300ms"
    class="btn btn-danger btn-sm">Yes, delete</button>
  <button
    hx-get="/research/{{ id }}/row"
    hx-target="#record-row-{{ id }}"
    hx-swap="outerHTML"
    class="btn btn-ghost btn-sm">Cancel</button>
</div>

Pattern 4 — Load more / infinite scroll

Used in: public research overview. The server returns the next page of cards. The "Load more" button replaces itself with the next page's button.

html — load more button
<!-- Results container -->
<div id="results" role="list" aria-live="polite" aria-label="Research results">
  <!-- Initial page of cards -->
</div>

<!-- Load more button — replaces itself with next page's button -->
<div id="load-more">
  <button
    class="btn btn-secondary"
    hx-get="/research?page=2&q={{ query }}"
    hx-target="#results"
    hx-swap="beforeend"
    hx-indicator="#load-spinner">
    Load more
    <span id="load-spinner" class="htmx-indicator ms-1">
      <span class="spinner-border spinner-border-sm" role="status" aria-label="Loading…"></span>
    </span>
  </button>
</div>

<!-- Server appends to #results AND replaces #load-more via OOB swap -->
<!-- Use hx-swap-oob="true" on the new load-more element in the response -->
html — server partial (/research?page=2 response)
<!-- These cards get appended to #results -->
<article class="card card-research" role="listitem">…</article>
<article class="card card-research" role="listitem">…</article>
<article class="card card-research" role="listitem">…</article>

<!-- OOB swap replaces the Load more button -->
<div id="load-more" hx-swap-oob="true">
  <button
    class="btn btn-secondary"
    hx-get="/research?page=3&q={{ query }}"
    hx-target="#results"
    hx-swap="beforeend">
    Load more
  </button>
</div>
<!-- If no more pages, return an empty #load-more div -->

Pattern 5 — Lazy-loaded detail sections

Used in: research detail page — related output, citation metrics. Load expensive sections only when the user scrolls to them.

html
<section aria-labelledby="related-heading">
  <h2 id="related-heading">Related research output</h2>

  <!-- hx-trigger="revealed" fires when element enters viewport -->
  <div
    hx-get="/research/{{ id }}/related"
    hx-trigger="revealed"
    hx-swap="innerHTML"
    hx-indicator="#related-spinner">

    <span id="related-spinner">
      <span class="spinner-border spinner-border-sm" role="status" aria-label="Loading related output…"></span>
    </span>
  </div>
</section>

Pattern 6 — Filter facets with URL persistence

Toggling a facet updates the results AND the URL, so the filtered view is bookmarkable.

html — facet checkbox
<!-- Each facet is a form that posts to the search endpoint -->
<!-- hx-push-url keeps the URL in sync with state -->
<form
  hx-get="/research"
  hx-target="#results"
  hx-swap="innerHTML"
  hx-push-url="true"
  hx-trigger="change"
  id="facet-form">

  <!-- Hidden: carry forward other active filters -->
  <input type="hidden" name="q" value="{{ current_query }}">

  <div class="c-facet-title">Type</div>
  <div class="c-facet-check">
    <input class="form-check-input mt-0" type="checkbox"
      id="f-journal" name="type" value="journal-article"
      {{ 'checked' if 'journal-article' in selected_types }}>
  </div>
  <div class="c-facet-name">
    <label for="f-journal">Journal article</label>
  </div>
  <div class="c-facet-count">1,234</div>

  <!-- More facets… -->
</form>

Accessibility rules for HTMX

Never do this

  • Swap content into a container without notifying screen readers — use aria-live
  • Move focus silently after a swap (e.g. a form submit that scrolls to errors)
  • Rely on HTMX for navigation without also supporting regular links
  • Use hx-boost on elements that aren't links or forms

Always do this

  • Wrap dynamic regions in aria-live="polite" (results, status messages)
  • Use aria-live="assertive" only for errors that need immediate attention
  • After a form error swap, move focus to the error summary: htmx.on('htmx:afterSwap', ...)
  • Show htmx-indicator spinners with proper role="status" and aria-label
  • Test keyboard navigation after every dynamic swap
  • Use hx-push-url when the user would expect Back to work
javascript — focus management after swap
// Move focus to first error after a failed form submission
document.body.addEventListener('htmx:afterSwap', function(evt) {
  const firstError = evt.detail.target.querySelector('.is-invalid, [role="alert"]');
  if (firstError) {
    firstError.focus();
    // If the element isn't focusable, focus its label instead:
    // document.querySelector(`[for="${firstError.id}"]`)?.focus();
  }
});

What HTMX is not for

HTMX handles server-driven interactions well. It is not the right tool for:

  • Real-time collaboration (multiple users editing the same record simultaneously) — use WebSockets or SSE
  • Complex client-side state (drag-and-drop reordering with optimistic UI) — consider a small Alpine.js component
  • File upload progress — the browser's native progress API or a dedicated upload library is more appropriate
  • Very high-frequency updates (notifications polling faster than 30s) — use SSE (hx-sse extension)

When you reach for JavaScript beyond HTMX, keep it small, isolated, and framework-free. Vanilla JS or Alpine.js only.
