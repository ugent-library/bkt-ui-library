Surfaces

The same design system serves two fundamentally different contexts. One faces readers, researchers, and the world. One faces curators, librarians, and depot managers. They share tokens, components, and HTML structure. What changes is the personality — applied by a single attribute on the layout container.

How to use it

Apply the attribute to the outermost layout element. Everything inside inherits the surface variables. You can nest surfaces — a backoffice shell can contain a public-facing preview.

html — public layout
<body data-surface="public">
  <!-- all components render with serif headings, generous spacing -->
</body>
html — backoffice layout
<body data-surface="backoffice">
  <!-- all components render compact, sans headings -->
</body>
html — mixed: backoffice with public preview panel
<body data-surface="backoffice">
  <!-- curator toolbar and record form -->
  <div class="bt-toolbar">...</div>

  <!-- live public preview of how the record will appear -->
  <aside data-surface="public" class="record-preview">
    <article class="card">...</article>
  </aside>
</body>
css — reading surface variables in a custom component
/* Component reads from surface — works in both contexts */
.my-component-title {
  font-family: var(--s-heading-font);      /* serif on public, sans on backoffice */
  font-weight: var(--s-heading-weight);
  letter-spacing: var(--s-heading-tracking);
}

.my-component-body {
  font-size: var(--s-body-size);           /* 16px public, 15px backoffice */
  line-height: var(--s-body-leading);      /* 1.7 public, 1.5 backoffice */
  padding: var(--s-card-pad);              /* 1.5rem public, .875rem 1rem backoffice */
}

Surface decision guide

?
The user is reading or discovering content. → Public surface. The content is the product.
?
The user is managing, curating, or operating on records. → Backoffice surface. Speed and overview are the product.
?
The user is depositing their own research output. → Public surface. This is a researcher's interaction with their own work, not a curation workflow. It should feel like the public face, not the backoffice.
?
The interface serves multiple user types on the same screen. → Use the surface of the primary user. A librarian reviewing a researcher's submission is doing curation work → backoffice.
?
depotmanagement (physical depot operations). → Backoffice. Dense. Fast. The worker is scanning, picking, moving — not reading.

Icon system

Two icon systems, one role each. They are not interchangeable.

UGent icon font — i.if.if-[name]

All functional UI icons. Actions, navigation, status, form decorations. Your team extends this set when you need new functional icons.

  • Button icons: download, upload, edit, delete, close, add
  • Navigation: home, back, forward, menu, settings
  • Status: check, warning, error, info, lock, unlock
  • Custom domain icons your team has added

Remix Icons — .ri-[name]

Decorative and graphic use only. Structural marks, illustration accents, empty state graphics. Never in buttons, labels, or status indicators.

  • Empty state illustrations
  • Decorative section marks (◎ as SVG)
  • Large graphic icons in onboarding
  • Where a visual accent is needed, not a functional signal

Accessible icon usage — three patterns

✓ Icon + visible text (most common)

Icon is decorative. aria-hidden="true" hides it from screen readers. The visible text carries the meaning.

html
<button type="button" class="btn btn-primary">
  <i class="if if-download" aria-hidden="true"></i>
  Download PDF
</button>

✓ Icon alone — label on the button

The aria-label goes on the button, never on the <i>. Screen readers announce the button's label, then its role ("Close dialog, button"). VoiceOver and NVDA both read this correctly.

html
<!-- ✓ Correct: aria-label on the button -->
<button type="button" class="btn btn-ghost" aria-label="Close dialog">
  <i class="if if-close" aria-hidden="true"></i>
</button>

<!-- ✗ Wrong: aria-label on the icon -->
<button type="button" class="btn btn-ghost">
  <i class="if if-close" aria-label="Close"></i>  <!-- unreliable -->
</button>

✓ Status icon alongside text

When the icon reinforces visible text, hide it. The text is sufficient.

html
<span class="badge-oa">
  <i class="if if-unlock if--xs" aria-hidden="true"></i>
  Open access
</span>

<!-- If the icon IS the only indicator, wrap with a span -->
<span aria-label="Open access">
  <i class="if if-unlock" aria-hidden="true"></i>
</span>
